# KymFlow macOS PKG Installer — Developer README

_Last updated: 2026-03-28T23:58:22.077361+00:00_

## Purpose and scope

This document describes the current KymFlow macOS installer implementation in `kymflow/packaging/macos-pkg/`.

It is based only on the current source-of-truth files provided for this build system:

- `build_pkg.sh`
- `postinstall.sh`
- `smoke_test.sh`
- `notarize_pkg.sh`
- `Distribution.xml`
- `resources/welcome.html`
- `resources/conclusion.html`
- `resources/license.txt`

This README is for developer maintenance of the installer pipeline, not for end users.

---

## Current status

The current installer system supports:

- building a staged payload from the current repo
- filtering common junk files from packaged content
- creating a component package with `pkgbuild`
- creating a final installer package with `productbuild`
- signing the final installer package with a **Developer ID Installer** certificate
- notarizing and stapling the final installer package in a separate step
- installing runtime files under:
  - `~/Library/Application Support/kymflow-pkg`
- installing a user workspace under:
  - `~/Documents/KymFlow`
- creating a launcher that opens Jupyter rooted at:
  - `~/Documents/KymFlow`
- tracking install state and writing persistent install logs
- verifying installation with `smoke_test.sh`

```text
      _secrets.example.sh        # optional committed template

        license.txt
      scripts/
        postinstall.sh
      payload/                   # generated by build_pkg.sh
      build/                     # generated by build_pkg.sh
      dist/                      # generated by build_pkg.sh
```

Notes:

- `payload/`, `build/`, and `dist/` are generated directories.
- `_secrets.sh` is local-only and should not be committed.
- `Distribution.xml` and `resources/` define the installer GUI flow used by `productbuild`.

## High-level architecture

### Build-time flow

`build_pkg.sh` does the following:

1. resolves the repo root and source paths
2. reads version from `pyproject.toml`
3. stages packaging payload under `payload/kymflow`
4. filters unwanted files during staging
5. prepares `pkgbuild` scripts by copying `postinstall.sh` to a file named `postinstall`
6. builds a component package with `pkgbuild`
7. builds a final signed distribution package with `productbuild`
8. verifies the final package signature with `pkgutil --check-signature`

### Install-time flow

`postinstall.sh` runs inside the installer and does the following:

1. determines the install user and user home directory
2. sets up runtime under `~/Library/Application Support/kymflow-pkg`
3. sets up persistent logs under `~/Library/Application Support/kymflow-pkg/logs`
4. reads packaged version from payload `pyproject.toml`
5. compares against installed version if present
6. determines install mode:
   - `first_install`
   - `reinstall_same`
   - `upgrade`
7. copies the payload into app support using filtered `rsync`
8. bootstraps `uv` if needed
9. installs Python if needed
10. creates or reuses a virtual environment
11. installs JupyterLab, ipykernel, and local `kymflow` when required
12. refreshes installer-managed workspace directories
13. creates the launcher script
14. fixes workspace ownership and permissions
15. writes `install_version.txt` and `install_state.json`

### Notarization flow

`notarize_pkg.sh` is a separate step and does the following:

1. loads local secrets
2. finds the newest `dist/KymFlow-*.pkg`
3. submits it to Apple notarization with `notarytool`
4. staples the result with `stapler`
5. validates stapling and Gatekeeper assessment

This is intentionally separate from `build_pkg.sh` so that build/sign and notarization remain distinct responsibilities.

## Detailed file: `build_pkg.sh`

### Purpose

Create the signed final `.pkg` installer.

### Inputs

- repo files from the current `kymflow` source tree:
  - `pyproject.toml`
  - `README.md`
  - `LICENSE`
  - `src/`
  - `notebooks/`
- local packaging files:
  - `scripts/postinstall.sh`
  - `Distribution.xml`
  - `resources/welcome.html`
  - `resources/conclusion.html`
  - `resources/license.txt`
  - `_secrets.sh`

### Required local secrets

`build_pkg.sh` requires a local file:

```text
kymflow/packaging/macos-pkg/_secrets.sh
```

It must define:

```bash
export SIGN_INSTALLER_ID='Developer ID Installer: Robert Cudmore (794C773KDS)'
export NOTARY_PROFILE='my-notarytool-profile-feb2025'
```

Only `SIGN_INSTALLER_ID` is required by `build_pkg.sh`.
`NOTARY_PROFILE` is used later by `notarize_pkg.sh`.

### Behavior, step by step

#### 1. Resolve core paths

The script computes:

- `SCRIPT_DIR`
- `ROOT_DIR`
- `REPO_ROOT`
- `KYMFLOW_SRC`
- `PAYLOAD_DIR`
- `BUILD_DIR`
- `DIST_DIR`
- `RESOURCES_DIR`
- `DIST_XML`

#### 2. Resolve package version

The script reads `pyproject.toml` and extracts:

- `version = "..."`

This becomes the package version and is used in the final filename:

```text
dist/KymFlow-<version>.pkg
```

#### 3. Validate required inputs

The script hard-fails if required files or directories are missing.

#### 4. Load local secrets

The script requires `_secrets.sh` and sources it.

It then requires:

- `SIGN_INSTALLER_ID`

#### 5. Stage payload

The payload is staged under:

```text
payload/kymflow/
```

Files copied:

- `pyproject.toml`
- `README.md`
- `LICENSE`
- `src/`
- `notebooks/`

#### 6. Exclude unwanted files

During `rsync`, the script excludes:

- `.DS_Store`
- `.ipynb_checkpoints`
- `__pycache__/`
- `*.pyc`

This is currently the deliberate payload cleanup boundary.

#### 7. Prepare `pkgbuild` scripts

`pkgbuild` expects installer scripts to be named exactly `postinstall` or `preinstall`.

The current source file is:

```text
scripts/postinstall.sh
```

So `build_pkg.sh` copies it into a generated scripts directory as:

```text
build/pkgbuild-scripts/postinstall
```

and marks it executable.

This is why the project source file can be called `postinstall.sh` while `pkgbuild` still receives a correctly named `postinstall`.

#### 8. Build component package

The script runs `pkgbuild` with:

- `--root payload/`
- `--scripts build/pkgbuild-scripts`
- `--identifier org.cudmore.kymflow`
- `--version <pkg version>`
- `--install-location /Library/Application Support/KymFlowPayload`

Output:

```text
build/KymFlowComponent.pkg
```

#### 9. Build signed final package

The script runs `productbuild` with:

- `--distribution Distribution.xml`
- `--resources resources/`
- `--package-path build/`
- `--sign "$SIGN_INSTALLER_ID"`

Output:

```text
dist/KymFlow-<version>.pkg
```

This is the main final installer artifact.

#### 10. Verify signature

The script runs:

```bash
pkgutil --check-signature dist/KymFlow-<version>.pkg
```

to verify the installer signature.

### Expected generated tree after running `build_pkg.sh`

```text

  payload/
    kymflow/
      LICENSE
      README.md
      pyproject.toml
      src/
      notebooks/
```

## Detailed file: `postinstall.sh`

### Purpose

Perform install-time runtime setup and workspace setup.

### Key paths

#### App support root

```text
~/Library/Application Support/kymflow-pkg
```

#### Workspace root

```text
~/Documents/KymFlow
```

### Files and directories under app support

Expected structure:

```text
~/Library/Application Support/kymflow-pkg/
  uv/
  venv/
  payload/
  logs/
  install_version.txt
  install_state.json
```

#### `uv/`

Purpose:

- local `uv` runtime installed by the installer if missing

#### `venv/`

Purpose:

- Python virtual environment used for JupyterLab and local `kymflow`

#### `payload/`

Purpose:

- copied runtime payload used by postinstall for local installation
- current staged content includes the packaged `kymflow` tree copied from `/Library/Application Support/KymFlowPayload/kymflow`

#### `logs/`

Purpose:

- persistent installer logs

File pattern:

```text
install-YYYYMMDD-HHMMSS.log
```

#### `install_version.txt`

Purpose:

- stores the installed package version as a single line

#### `install_state.json`

Purpose:

- machine-readable install state

Current structure:

```json
{
  "version": "...",
  "python_version": "...",
  "install_mode": "...",
  "log_file": "...",
  "updated_at_utc": "..."
}
```

### Behavior, step by step

#### 1. Determine current user and home directory

The script resolves:

- `CURRENT_USER`
- `USER_HOME`

It falls back to `dscl` if `HOME` is not usable.

#### 2. Create app support root and log directory

Creates:

- `APP_ROOT`
- `LOG_DIR`

Then it opens a timestamped log file and tees stdout/stderr into it.

#### 3. Validate packaged payload

The script expects the installer payload to have been laid down under:

```text
/Library/Application Support/KymFlowPayload/kymflow
```

It validates required files and directories there before proceeding.

#### 4. Resolve packaged version and installed version

- packaged version is read from packaged `pyproject.toml`
- installed version is read from `install_version.txt` if present

#### 5. Determine install mode

Possible modes in current script:

- `first_install`
- `reinstall_same`
- `upgrade`

There is also explicit downgrade-blocking logic in the script.
This README describes that as script behavior; whether a downgrade has been manually tested is a separate question.

#### 6. Prepare workspace base directories

Creates the workspace root and relevant directories.

#### 7. Copy payload into app support

The script refreshes the app-support payload by `rsync` from the packaged payload root into:

```text
~/Library/Application Support/kymflow-pkg/payload/kymflow
```

It excludes:

- `.DS_Store`
- `.ipynb_checkpoints`
- `__pycache__/`
- `*.pyc`

#### 8. Bootstrap `uv`

If `uv` is not already present, it is installed under:

```text
~/Library/Application Support/kymflow-pkg/uv
```

#### 9. Ensure Python and venv

- installs Python version from `INSTALL_PYTHON_VERSION` if needed
- creates venv if missing
- reuses venv if present

#### 10. Runtime install / reinstall logic

For same-version reinstall:

- if the existing venv is healthy and imports the packaged `kymflow` version successfully, runtime reinstall is skipped

Otherwise:

- install `jupyterlab`
- install `ipykernel`
- install local `kymflow` from app-support payload
- register Jupyter kernel

#### 11. Manage workspace content

Current workspace model:

```text
~/Documents/KymFlow/
  Examples/
  Example-Data/
  User/
  Open KymFlow.command
```

##### `Examples/`

Purpose:

- installer-managed examples
- refreshed on every install

Behavior:

- removed and recreated
- repopulated from packaged notebooks

##### `Example-Data/`

Purpose:

- installer-managed example data
- currently created and refreshed to the managed empty state

Behavior:

- removed and recreated
- currently left empty because no packaged example-data source has been added yet

##### `User/`

Purpose:

- user-owned working directory

Behavior:

- created if missing
- not overwritten by reinstall

#### 12. Create launcher

Creates:

```text
~/Documents/KymFlow/Open KymFlow.command
```

Behavior:

- launches Jupyter using the venv Jupyter binary
- roots Jupyter at:
  - `~/Documents/KymFlow`

This is important: the launcher does **not** root Jupyter at `Examples/`, `User/`, or `Notebooks/`.

#### 13. Fix permissions

The script applies:

- `chown -R CURRENT_USER:staff`
- `chmod -R u+rwX`

to the workspace root so user files are writable.

#### 14. Write install state

Writes:

- `install_version.txt`
- `install_state.json`

## Detailed file: `notarize_pkg.sh`

### Purpose

Perform notarization and stapling as a separate step after `build_pkg.sh`.

### Why separate from `build_pkg.sh`

This keeps responsibilities clean:

- `build_pkg.sh`:
  - build
  - sign
  - verify signature

- `notarize_pkg.sh`:
  - submit to Apple
  - staple
  - validate notarization result

### Inputs

- local `_secrets.sh`
- newest matching package in:
  - `dist/KymFlow-*.pkg`

### Required local secrets

Requires:

```bash
export NOTARY_PROFILE='my-notarytool-profile-feb2025'
```

### Behavior

1. load `_secrets.sh`
2. require `NOTARY_PROFILE`
3. locate newest package in `dist/`
4. submit with:
   ```bash
   xcrun notarytool submit <pkg> -p "$NOTARY_PROFILE" --wait
   ```
5. staple with:
   ```bash
   xcrun stapler staple <pkg>
   ```
6. validate stapled ticket with:
   ```bash
   xcrun stapler validate <pkg>
   ```
7. check Gatekeeper assessment with:
   ```bash
   spctl -a -vv --type install <pkg>
   ```

### Output

The package remains in place in `dist/`; the script updates it in place by stapling the notarization ticket.

## Detailed file: `smoke_test.sh`

### Purpose

Quickly verify a local install is structurally correct.

### Current checks

The script verifies:

- venv Python exists
- venv Jupyter exists
- `install_state.json` exists
- logs directory exists
- at least one install log file exists
- workspace directories exist:
  - `Examples/`
  - `Example-Data/`
  - `User/`
- launcher exists
- `import kymflow` works
- Jupyter reports a version

### Expected pass conditions

If the install completed correctly and postinstall ran successfully, `smoke_test.sh` should pass without manual environment activation.

## Signing and notarization model

### Current package signing

The installer package is signed with a **Developer ID Installer** identity through `productbuild --sign`.

The current expected local secret value is:

```bash
SIGN_INSTALLER_ID='Developer ID Installer: Robert Cudmore (794C773KDS)'
```

### Do the `.py` files need separate signing?

Not in the current packaging model.

The current distribution flow packages Python source files and notebooks into a signed installer package. There is no `.app` bundle or embedded signed executable artifact being prepared in this flow, so there is no current additional step to sign `.py` files with **Developer ID Application**.

A separate **Developer ID Application** identity would become relevant later if the project starts distributing things like:

- `.app` bundles
- helper executables
- native binaries
- embedded frameworks

For the current installer flow, the critical signing step is:

- final `.pkg` signed with **Developer ID Installer**

### Verification commands

Signed package verification:

```bash
pkgutil --check-signature dist/KymFlow-<version>.pkg
```

Stapling validation:

```bash
xcrun stapler validate dist/KymFlow-<version>.pkg
```

Gatekeeper install assessment:

```bash
spctl -a -vv --type install dist/KymFlow-<version>.pkg
```
